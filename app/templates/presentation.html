Introduction

What is A2Z?

A highly secure, zero-knowledge platform designed to safely store and share credentials across an organization.

Zero Trust

The server never sees your master password or plaintext data.

Org Sharing

Department-level credential sharing with RSA-OAEP key wrapping.

TOTP 2FA

Multi-factor authentication for defense-in-depth security.

Methodology: Key Derivation

Argon2id Client-Side Derivation

The user's Master Password is processed through Argon2id (memory-hard, 64KB memory cost) entirely in the browser. It produces a 64-byte combined key split into two:

AuthKey (32 bytes)

Sent to the server for authentication. Never stored in plaintext - server applies PBKDF2-HMAC-SHA256 (200K iterations) before storage.

MasterEncryptionKey (32 bytes)

Never leaves the browser. Held temporarily in RAM only. Used for AES-256-GCM vault encryption/decryption. Cleared on logout.

Methodology: Encryption

AES-256-GCM Authenticated Encryption

Data confidentiality and integrity via authenticated encryption. The server stores only ciphertext blobs and IVs.

crypto.js
async function encryptAesGcm(plaintext, key) {
  const iv = crypto.getRandomValues(new Uint8Array(12));
  const ciphertext = await subtle.encrypt(
    { name: 'AES-GCM', iv, tagLength: 128 },
    key, encode(JSON.stringify(plaintext))
  );
  return { iv: toBase64(iv), ciphertext: toBase64(ciphertext) };
}

Tamper detection: any modified ciphertext fails the GCM auth tag check.

RSA-OAEP (4096-bit) wraps shared keys for department credential sharing.

Methodology: Server-Side

PBKDF2 Double Hashing

The server applies PBKDF2-HMAC-SHA256 on the incoming AuthKey with a random 16-byte salt and 200,000 iterations. This stops Pass-the-Hash attacks.

auth_utils.py
def hash_auth_key(auth_key_b64, salt=None):
    if salt is None:
        salt = secrets.token_bytes(16)
    derived = hashlib.pbkdf2_hmac(
        "sha256", auth_key_b64.encode(),
        salt, 200_000
    )
    return f"{salt.hex()}${derived.hex()}"

Security: Transport Layer

Nginx Transport Layer Security

Securing the communication channel is vital. The application is deployed behind an Nginx reverse proxy configured with strict security measures.

Strict HTTPS

Mandates HTTPS by permanently redirecting all HTTP traffic. Strictly allows only TLSv1.2 and TLSv1.3.

HSTS & Isolation

Strict-Transport-Security (HSTS) ensures secure connections, while X-Frame-Options prevents clickjacking.

COOP/CORP Policies

Isolates the browsing context to protect the ephemeral MasterEncryptionKey from cross-site leaks.

Security: Organizational

Administrative & Organizational Security

Administrative functions are secured through rigorous backend validation and session management to prevent unauthorized privilege escalation.

Department Assignment: Securely authenticated and authorized before any organizational state changes occur.

Service Creation: Prevents unauthorized privilege escalation via strict backend validation.

Future Work

Mitigating Memory Scraping

Web browsers lack OS-level memory locking, leaving keys theoretically vulnerable to memory scraping. Future enhancements include:

Native Client

Transitioning to a native application to utilize OS-level memory locking APIs.

WebAssembly (Wasm)

Provides finer control over memory, allowing instant zeroing of key buffers.

Hardware Crypto

Integrating with TPMs or physical security keys via WebAuthn to keep keys out of RAM entirely.

Conclusion

A2Z successfully implements a highly secure, zero-trust architecture. Client-side Web Crypto, Argon2, and AES-GCM protect data even in a catastrophic server breach.

DB Leak Proof

Argon2 + PBKDF2 layering means a database breach yields zero usable credentials.

Network Safe

Protected by Strict TLS 1.3, HSTS, and Nginx isolation headers. Interception yields only AES-GCM ciphertext.

Integrity

GCM auth tags detect any tampered ciphertext, alerting the client immediately.
